Documentation Extractor

Enter any domain, discover its XML sitemaps, and extract every documentation URL — filtered and ready to paste into NotebookLM or any AI knowledge tool. Turn any doc site into an instant expert guide.

The Problem

LLMs hallucinate often, especially when asked for information on complex and frequently updated technical documentation. Ask about a specific API parameter, CLI flag, or configuration schema and there's a good chance it'll confidently give you something that doesn't exist. The training data is often outdated or generic — and for technical documentation, "close enough" isn't good enough.

The fix is grounding the model with official, up-to-date documentation so it has the real source of truth. Tools like NotebookLM make this easy — if you have the URLs. But most documentation sites have hundreds or thousands of pages spread across nested sitemaps. Manually hunting for the right URLs is tedious and incomplete.

I kept running into this friction every time I wanted to create an AI-powered reference for a new technology. The Documentation Extractor was built to eliminate it.

The Solution

Enter a domain and the Documentation Extractor discovers its sitemaps — checking robots.txt for directives and falling back to common paths. It recursively parses every sitemap index, fetches URLs in batches with exponential-backoff retry, and deduplicates the results.

A keyword filtering UI lets you narrow results in real time — include or exclude URL paths with a tag-bubble interface — then copy the filtered list to your clipboard with one click. Paste directly into NotebookLM and you've got an expert guide grounded in official sources, built in seconds instead of hours.

Limitations

The Documentation Extractor was built as a lightweight prototype, and it has several limiting constraints.

  • Requires a well-structured sitemap. The tool relies on a valid robots.txt pointing to XML sitemaps, or a /sitemap.xml at the root. Sites without this setup return no results.
  • Large sitemaps time out. Google Apps Script has a 6-minute execution limit. Sites with very large sitemaps (e.g., cloud.google.com with 100,000+ URLs) will exceed this, causing the app to fail. It works best on small-to-medium documentation sites.
  • No progress feedback. Because google.script.run is a single async call, there's no way to stream progress back to the UI. The spinner shows until processing either completes or times out.
  • Apps Script quota limits. UrlFetchApp is subject to daily quota limits. Heavy use may trigger rate limiting even with exponential backoff.
  • NotebookLM source limits. NotebookLM caps sources at 50 or 300 depending on your Google account tier. The keyword filtering helps you stay within that limit, but it significantly constrains the depth and coverage of the guides you can build from large documentation sites.
  • Native documentation chatbots. Many large tech companies now embed chatbot assistants directly into their documentation sites, which can be more reliable for single-site Q&A.

For a more fully featured version that addresses some of these constraints, see DocWeb.

Where It Fits

The Documentation Extractor is the "v0" prototype that proved out a core workflow: automated sitemap discovery and URL extraction. That workflow became the starting point for DocWeb, a more fully featured sitemap extraction and chatbot platform that addresses many of these limitations.
